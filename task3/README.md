# Задание №3 (1000 серверов)

* **Алгоритм честного Round-Robin:** Скрипт сортирует серверы по Unix-timestamp последней успешной заливки. В каждый час выбирается самый "старый" (или еще ни разу не обновлявшийся) сервер, что гарантирует равномерное распределение нагрузки.
* **Атомарность передачи данных:** Использование `rsync` с флагом `--delay-updates` гарантирует, что файл на сервере заменяется мгновенно только после полной загрузки.
* **Отказоустойчивость (Failover):** Если приоритетный сервер по очереди недоступен (упал SSH или таймаут соединения), скрипт автоматически переходит к следующему в очереди, пока не найдет «живой» хост.
* **Автоматическое создание путей:** Использование `--mkpath` в `rsync` предотвращает сбои, если на целевом сервере отсутствует нужная директория для сохранения.
* **Автоматизация инфраструктуры:** В состав проекта включен Ansible-плейбук для массовой подготовки целевых хостов.

## Инструкция по развертыванию и использованию

### Шаг 1. Настройка беспарольного доступа
Убедитесь, что публичный SSH-ключ пользователя root с управляющего сервера скопирован на все 1000 серверов в файл /root/.ssh/authorized_keys.

### Шаг 2. Подготовка хостов через Ansible
С управляющего сервера запустите плейбук для быстрой параллельной установки rsync на весь инвентарь (флаг -f 50 запускает процесс в 50 параллельных потоков):

```
cd ansible
ansible-playbook install_rsync.yml -f 50
cd ..
```

### Шаг 3. Деплой рабочих скриптов
Создайте системную директорию согласно стандарту FHS и перенесите туда файлы бизнес-логики:

```
sudo mkdir -p /opt/orchestrator
sudo cp task3/upload.py /opt/orchestrator/
sudo cp task3/hosts.txt /opt/orchestrator/
sudo cp task3/payload.txt /opt/orchestrator/
```

`Предоставление прав на исполнение управляющему скрипту`
```
sudo chmod +x /opt/orchestrator/upload.py
```
### Шаг 4. Активация расписания в Cron
Зарегистрируйте задачу в планировщике операционной системы:

```
sudo cp task3/file_uploader /etc/cron.d/file_uploader
sudo chmod 644 /etc/cron.d/file_uploader
sudo chown root:root /etc/cron.d/file_uploader
```

`Принудительное обновление конфигурации демона cron`
sudo systemctl reload cron

## Ответы на вопросы из задания
### Как это можно сделать?
```
Для этого применяется алгоритм Round-Robin с сохранением состояния. Каждый час скрипт, запускаемый по Cron, выстраивает очередь из серверов на основе их последнего обновления. Сервер со старейшим таймстампом выбирается главным кандидатом на загрузку. После успешной передачи его таймстамп обновляется на текущее время, и он перемещается в самый конец очереди. Это гарантирует, что каждый из 1000 серверов получит файл ровно 1 раз за 1000 часов, исключая повторные или холостые загрузки на один и тот же хост.
```

### Какие частные случаи следует учитывать?

1) Недоступность целевого сервера:
```
Решение: Внедрена функция пред-проверки check_ssh с коротким таймаутом ConnectTimeout=5. Если приоритетный сервер в очереди не отвечает, скрипт логирует событие, пропускает хост и переходит к следующему "старому" серверу. Часовое окно доставки не теряется.
```

2) Отсутствие целевой директории на удаленной машине:
```
Решение: В команду rsync интегрирован флаг --mkpath, который автоматически инициализирует поведение mkdir -p для всей структуры удаленного пути прямо перед записью файла.
```

3) Чтение файла сторонним приложением в момент загрузки:
```
 Реализована атомарность доставки с помощью флага --delay-updates. Утилита rsync скачивает данные во временный скрытый файл-буфер, проверяет контрольную сумму и только при 100% успехе мгновенно заменяет старый файл на новый на системном уровне.
```

4) Проблема «Переполненного диска»
```
Решение: Благодаря флагу --delay-updates, rsync не затрет старый рабочий файл на сервере, если новый не поместился на диск. Файл не повредится. Однако в логи упадет ошибка, и этот сервер на следующем часу снова попытается обновиться.
```

5) Изменение IP-адресов
```
Решение: Скрипт защищен от этого благодаря флагу -o StrictHostKeyChecking=no. Если бы его не было, SSH увидел бы, что у старого IP изменился цифровой отпечаток и заблокировал подключение. С этим флагом скрипт просто обновит файл на новой машине.
```

6) «Эффект бабочки» при изменении системного времени
```
На управляющем сервере сбилось системное время. Сортировка по last_time сломается: серверы, которые обновились только что, получат таймстамп из «будущего» и надолго застрянут в самом конце очереди, нарушая логику Round-Robin.
Решение: В идеальной инфраструктуре на управляющем сервере всегда должен быть активен демон синхронизации времени, который жестко держит эталонное время по UTC.
